## What This Project Is

A **Developer Assessment Platform** that enables companies to register candidates, build question banks, generate tailored technical assessments, distribute them securely via email, and evaluate submissions through automated MCQ marking and manual marker review.

---

## Roles

| Role | Responsibilities |
|---|---|
| **Admin** | Registers candidates and markers, manages assessments, views all results |
| **Marker / Reviewer** | Manages question banks, assigns assessments, marks submissions, sends results |
| **Candidate** | Receives invitation, completes assessment within the time limit, views own results |

Candidates are **always registered by an Admin or Marker** — there is no self-registration in v1.

## Package Layout

### Backend

```
com.psybergate.dap
  ├── controller/     ← REST controllers; input validation and delegation only; no business logic
  ├── service/        ← All business logic and @Transactional boundaries
  ├── repository/     ← Spring Data JPA interfaces; custom @Query methods
  ├── domain/         ← JPA entities, enums, and domain exceptions; extends BaseEntity
  ├── dto/            ← Java Records for request/response; separate types per direction
  └── config/         ← Spring configuration (@Configuration classes: security, Liquibase, etc.)
```

Layer rules:
- `controller/` may only call `service/` — never `repository/` or `domain/` directly
- `service/` may call `repository/` and use `domain/` objects — never another service's repository
- `repository/` operates on `domain/` entities only
- `dto/` and `config/` are shared across layers but own no logic

### Frontend

```
src/app/
  core/          # singleton services, interceptors, guards
  shared/        # reusable components, pipes, directives
  features/
    feature-name/
      components/
      services/
      models/
      feature.routes.ts
```

## Domain Model (Key Entities)

| Entity | Notes |
|---|---|
| `app_user` | All roles in one table; discriminated by `role` enum (`ADMIN`, `MARKER`, `CANDIDATE`) |
| `candidate` | 1-to-1 with `app_user`; `candidate_id` is also the FK to `app_user` |
| `question_bank` | Named category that owns questions |
| `assessment_question` | Abstract; `TABLE_PER_CLASS` inheritance → `mcq_question`, `text_question`, `doc_question`, `question_group` |
| `assessment` | Assigned to one candidate; holds `invitation_token`, `status`, `time_limit_minutes`, `start_time` |
| `submission` | One per assessment (enforced by `UNIQUE` constraint on `assessment_id`) |
| `response` | Abstract; `TABLE_PER_CLASS` → `mcq_response`, `text_response`, `doc_response`, `question_group_response` |

All PKs are UUIDs generated by the database (`gen_random_uuid()`), never by Java.

## Critical Business Rules

### No-Repeat Questions
When generating an assessment, the system must exclude any question the candidate has previously seen across **all** of their past submitted assessments. This check runs at generation time.

### One Submission Only
A candidate may not retake an assessment. The `UNIQUE` constraint on `submission(assessment_id)` enforces this at the DB level. The service layer must also check `status = SUBMITTED` before accepting a submission request.

### Doc Question Limit
At most **one** `doc_question` per assessment. Enforced in the service layer at assessment creation time.

### Server-Side Timer
The submission endpoint checks `now() > start_time + time_limit_minutes`. If the time has expired, the server auto-submits whatever responses have been saved and sets `auto_submitted = true`. The frontend countdown is a convenience only — the server is the authority.

### Assessment Status Machine
```
PENDING → IN_PROGRESS → SUBMITTED → MARKED
```
- `PENDING`: created and assigned
- `IN_PROGRESS`: candidate accessed the assessment via valid token
- `SUBMITTED`: candidate submitted OR server-side timer fired
- `MARKED`: marker completed all response marking

### Result Response Email
When a Marker finalises marking, the system sends the candidate a **response email** with the Marker's feedback and comments. The email does not contain scores or marks. Never send this email automatically on submission — it is only triggered once the Marker explicitly finalises.

### MCQ Auto-Marking
MCQ responses are auto-marked immediately on submission by comparing `selected_answers` against `correct_answers`. Both single and multiple correct answer MCQs are supported.

## API Conventions

- Base path: `/api`
- All paths are plural, lowercase, hyphen-separated nouns — no verbs in paths
- All IDs are UUIDs in string format
- All timestamps are ISO-8601 UTC
- Pagination: `page` (0-indexed), `size` (default 20, max 100), `sort`
- Error shape: `{ status, error, message, timestamp }`
- File uploads use `multipart/form-data`

### Auth
- All endpoints are protected **except** `POST /api/auth/login` and `GET /api/assessments/access/{token}`
- JWT payload: `sub` (user UUID), `role`, `exp`
- The `invitation_token` on an assessment is a **separate** short-lived signed JWT — it grants token-gated access to one assessment without requiring the candidate to be logged in first

## Coding Standards

### Backend
- Controllers: REST only — validate input, delegate to service, map to DTO. No business logic.
- Use `@Autowired` on constructors for dependency injection — never on fields
- Services: all business logic, all `@Transactional` annotations
- Repositories: Spring Data JPA interfaces; never called from outside their own layer
- Entities: extend `BaseEntity` (provides `id`, `createdAt`, `updatedAt`); no business logic; no Jackson annotations
- DTOs: Java Records; separate `Request` / `Response` types; `@Valid` on all controller parameters

### Lombok Rules
| Annotation | Rule |
|---|---|
| `@Getter`, `@Setter` | ✅ Preferred on entities; allowed on DTOs |
| `@Builder` | ✅ Allowed on entities and DTOs |
| `@NoArgsConstructor`, `@AllArgsConstructor` | ✅ Allowed on entities (required for JPA) |
| `@ToString` | ✅ Allowed — but use `@ToString.Exclude` on lazy associations |
| `@Data` | ❌ **Forbidden** — breaks JPA proxies |
| `@SneakyThrows` | ❌ **Forbidden** — never suppress checked exceptions silently |
| `@EqualsAndHashCode` | ⚠️ Only when explicitly needed; specify the ID field only |

### JPA Rules
- Every service method that navigates a lazy association must be `@Transactional(readOnly = true)`
- Lazy loading is the default for all associations — `FetchType.EAGER` is forbidden
- Use `@EntityGraph` or `JOIN FETCH` in repository queries when you need associations in one query
- Use `@Enumerated(EnumType.STRING)` for all enum columns
- Never use `CascadeType.REMOVE` unless the child is a true aggregate member

### Liquibase Rules
- One changeset per logical change
- ID format: `YYYY-MM-DD-NNN-short-description`
- Author field must be the developer's name
- Rollback blocks required on all destructive changesets
- Never edit or delete a changeset that has been applied to any shared environment

### Angular / TypeScript Rules
- Standalone components only — no NgModules
- `OnPush` change detection on all components
- `input()` signal API for component inputs — not `@Input()` decorator
- Reactive forms only — no template-driven forms
- `takeUntilDestroyed()` for subscription cleanup
- Never call the API directly from a component — always via a service
- `async` pipe in templates preferred over `.subscribe()` in components

## What Is Out of Scope (v1)

Do not build, suggest, or assume the following exist in v1:

- Candidate self-registration
- Automated code execution or sandboxed testing
- Cheating detection (tab switching, timing anomalies, similarity detection)
- AI-generated content detection
- Multi-language code runners (Java, Python, C#)
- OAuth2 / SSO — JWT only

The architecture should remain extensible for these but they must not be implemented now.
